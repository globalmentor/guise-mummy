- 2026-02-14 **Insight**: The "plan" produced by the PLAN phase is the site manifest—a complete inventory of artifacts with all metadata, not procedural instructions. The desired functionality is surfacing this existing data structure in human-readable form, not computing new information.
- 2026-02-14 **Finding**: The PLAN phase is both necessary and sufficient. Artifact metadata including `mummy/altLocation` redirects is loaded during planning (via `loadSourceMetadata()` parsing source files), not during mummification. The plan contains everything needed for site statistics and redirect inventory.
- 2026-02-14 **Decision**: Different commands can use different flag names for the same phase-bound functionality. Use `--describe` for `guise plan` (where the object is implicit) and `--describe-plan` for later-phase commands like `deploy` (where the plan must be named explicitly). This unblocked CLI design after incorrectly assuming all commands must share the same option name.
- 2026-02-14 **Decision**: Phase-bound execution model via `MummyExecution` enum. The describe behavior executes inside `GuiseMummy.mummify()` at the PLAN phase, controlled by passing `MummyExecution.DESCRIBE_PLAN` in a `Set<MummyExecution>` parameter. Analogous to Maven plugin executions bound to a lifecycle phase. Alternative of post-hoc accessor (`findPlan()` on `GuiseMummy`) rejected because describe is phase-bound behavior (fires after PLAN, before MUMMIFY), which is the engine's concern, not the CLI's.
- 2026-02-14 **Insight**: By defaulting `--describe` to `true` for `guise plan`, the command naturally becomes the site "info" command—running `guise plan` without flags prints the site composition summary. The flag exists to disable via `--describe=false` if silent planning is ever needed and for consistency with `--describe-plan` on other commands.
- 2026-02-14 **Decision**: Output goes to stdout, not the logger. This is user-requested information (the user asked for it via a flag), not logging or error output. Plain text format without ANSI terminal styling—deferred as gold-plating with no known need.
- 2026-02-15 **Insight**: The initial `PlanDescriber` implementation used ad-hoc string manipulation (`replace('\\', '/')`) for path display, violating the codebase's established path/URI model. Path display must follow the URI-domain processing chain: filesystem `Path` → `URI` (via `Path.toUri()`) → `URIPath` → resolve/relativize → display. Never string manipulation. This is the same pattern `S3Website.planResource()` uses for `altLocation` processing — the existing model already solved this problem.
- 2026-02-15 **Decision**: Deferred asset, veiled, collection-with-content, and subsumed artifact reporting due to model limitations. Asset status is transient (detected in `DirectoryMummifier.plan()` via name pattern but not persisted on `Artifact`). `Artifact.isNavigable()` incomplete for directories (always returns `true`). Phantom content files auto-generated for nearly all directories. Reasons documented in remediation plan to prevent re-research.
- 2026-02-25 **Finding**: `rootTargetPathUri` requires `toCollectionURI()` when calling `PlanDescriber` constructor. During PLAN phase the target directory doesn't exist on disk, so `Path.toUri()` omits the trailing slash (produces `file:///C:/project/target/site` instead of `file:///C:/project/target/site/`). Without `toCollectionURI()`, every `URIPath.relativize()` call treats `site` as a filename and relativizes against the parent — breaking all paths. `S3.plan()` doesn't need this because it runs during DEPLOY phase when directories exist. Lifecycle phase matters — established patterns can't be blindly followed.
- 2026-02-25 **Milestone**: Remediation plan fully specified (10 steps), all reviewer feedback incorporated, formatting discrepancies resolved, deferred categories documented with specific model limitations.
- 2026-02-25 **Decision**: CLI output uses `%n` format specifier for platform-native line endings, not hardcoded `\n`. On Windows, `> file.txt` redirection produces CRLF when the program emits `\r\n`. Using `\n` would produce Unix line endings in redirected output on Windows. `%n` is trivially easy and the standard Java idiom for this.
- 2026-02-26 **Insight**: `PlanDescriber.resourceReference()` and `PlanDescriber.collectRedirect()` appeared to be describer-local methods but were actually instances of a general pattern duplicated across three locations (`PlanDescriber`, `S3.plan()`, `AbstractMummyPlan.relativizeResourceReference()`). The core operation — relativize a URI against a base with optional collection form — belongs on `Artifact` as a static utility, not on any consumer. Factoring it out revealed a clean two-method layering: a pure URI-level `relativizeResourceReference(URI, URI, boolean)` (temporarily on `Artifact`, belongs in globalmentor-core) and an artifact-aware `relativizeResourceReference(URI, Artifact)` that derives `forceCollection` from `instanceof CollectionArtifact`.
- 2026-02-26 **Decision**: `collectRedirect()` made `static` with explicit `rootTargetPathUri` parameter, eliminating all instance field dependencies. Both it and `relativizeResourceReference` are now independently testable without constructing a `PlanDescriber`. The instance method `resourceReference()` was removed entirely — replaced by `Artifact.relativizeResourceReference(URI, Artifact)`.
