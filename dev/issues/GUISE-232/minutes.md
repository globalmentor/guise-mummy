- 2026-06-05 **Finding**: The XHTML parse error (`element type "p" must be terminated by </p>`) traces to `</code></pre>` landing on a single line after a blank line had already ended the outer `<div>` type-6 block. That line meets no HTML block start condition — type 6 rejects `code` (not in the enumerated list), type 7 rejects two tags on one line — so it becomes a Markdown paragraph, producing `<p></code></pre></p>`, which the XML parser rejects as unbalanced.
- 2026-06-05 **Insight**: CommonMark HTML blocks are lexical line-spans, not representations of HTML element trees. Block type is determined by the first tag on the freshly-evaluated start line; nesting is not tracked. A type-6 block (opened by `<div>`, `<details>`, etc.) terminates at the first blank line regardless of any tags nested inside it; a `<pre>` inside a type-6 block does not enter type-1 mode. Type-1 protection is only available when `<pre>` is itself the outermost freshly-evaluated start line.
- 2026-06-05 **Decision**: Resolved as `invalid` — Flexmark correctly implements the CommonMark spec; no parser bug or configuration option is needed. The fix is a Markdown source change: insert a blank line between the outer wrapper tag (making it a standalone one-line type-6 block) and `<pre><code>`, so `<pre>` is freshly evaluated and opens a type-1 block. A type-1 block tolerates internal blank lines and captures all content raw until a line containing `</pre>`, making `</code></pre>` on one line safe.
- 2026-06-05 **Finding**: Closing tags for outer HTML wrapper elements (e.g., `</figure>`, `</details>`) after a type-1 block closes must not be tab-indented. Post-close lines are freshly evaluated for start conditions; a leading tab counts as 4 columns, exceeding the 3-space maximum and demoting the line from an HTML block start to an indented code block — the tag text is then emitted as escaped literal content rather than raw HTML.
