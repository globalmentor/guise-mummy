# [GUISE-233] Minutes

- 2026-06-05 **Insight**: Guise Mesh has no existing facility — attribute, escape mechanism, or processing instruction — that means "don't process this region." The `^{…}` delimiter has no escape sequence; a literal `^{` always opens an expression. This rules out per-occurrence escaping as a solution for arbitrary inserted content.
- 2026-06-05 **Finding**: `mx:text` results are re-interpolated by `meshChildNodes()` due to evaluation order — `setTextContent()` creates a new text node, then the child-node walk scans it for `^{…}`. Inline `^{…}` results are not re-scanned. The asymmetry is undocumented and likely unintentional; it has correctness and latent security implications. Filed as a separate TODO for investigation and potential ticket.
- 2026-06-05 **Insight**: The distinction between "skip interpolation only" (Thymeleaf `th:inline="none"`) and "skip all processing" (Vue `v-pre`) is a real design axis. The `<li mx:each>` use case — iterate structurally while leaving text literal — demonstrates that disabling interpolation while keeping `mx:` directives active is a legitimate need, not just a theoretical possibility.
- 2026-06-05 **Finding**: Survey of templating systems confirms the industry converges on a negative-marker form (processing on by default, opt out for a region). No mainstream system uses a positive "templating-enabled" flag. The only enumerated approach is Thymeleaf's `th:inline`, and even that is a special case.
- 2026-06-05 **Insight**: Thymeleaf's `th:inline` enum conflates three concerns under one attribute: a scanning on/off switch (`none`), and a target-language selector (`text`/`javascript`/`css`) that governs value conversion. The `text`/`javascript`/`css` values select a *value-converter strategy* (how a Java object becomes characters for the target language), not an escaping strategy. Escaping is handled by the serializer, which is a separate layer. Twig's deliberate naming split — `verbatim` for region skip, `raw` for un-escape — is the principled alternative.
- 2026-06-05 **Insight**: Mesh processing decomposes into four independent axes: (1) structural/directive processing (`mx:each`, `mx:text`, `mx:attr-*`), (2) inline text processing (scanning `CharacterData` for `^{…}`), (3) value conversion (how expression results become text — plain `toString()` today; JS/CSS literal conversion hypothetically), (4) serialization (DOM to bytes, owned by `HtmlSerializer`, not Mesh). Any control mechanism should map cleanly onto one or more of these axes without conflating them.
- 2026-06-06 **Finding**: `HtmlSerializer` in globalmentor-web already disables text encoding for `<script>` element children via `isChildTextEncoded()`. This means if Mesh ever adds JS-aware value conversion, the serializer would pass the converter's output through unmolested in `<script>` — the two layers compose cleanly. Prior assumption that "XML serializers typically don't have this special case" was wrong.
- 2026-06-06 **Insight**: Mesh already has two implicit text-processing strategies: *template* (child text nodes are scanned for `^{…}`) and *expression* (`mx:text` evaluates its value as a MEXL expression). Both produce element text content, but via different processing paths. The feature being added is a third strategy: *literal* (pass through unchanged). Recognizing these as three values on a single "how is text content processed" axis unifies the existing behavior with the new feature.
- 2026-06-06 **Decision**: Rejected coupling the new attribute to `mx:text`. Initially considered `mx:text-mode` governing both `mx:text` attribute values and child content, with defaults of `expression` for `mx:text` and `template` for child content. Abandoned because it entangles two separate concerns: `mx:text` is an `mx:` directive (expression by definition, like all `mx:` attributes), while the feature needed is about element *content* processing. The reviewer identified multiple interaction gaps in the coupled design (what does `expression` mean for child content? what happens with inherited `literal` plus `mx:text`?).
- 2026-06-06 **Decision**: Adopted `mx:content-mode` with values `template` (default) and `literal` for the initial implementation. The attribute governs how Mesh interprets an element's textual content (child `CharacterData`). It does not affect `mx:` directive processing, `mx:text` evaluation, or non-`mx:` attribute interpolation. The name "content" refers to element content in the DOM sense, distinguishing it from attributes. The value `template` exists as an explicit override for re-enabling processing within an inherited `literal` region.
- 2026-06-06 **Decision**: `mx:content-mode` inherits down the DOM subtree. A descendant can override with an explicit `mx:content-mode="template"`. Initial implementation will use a simple ancestor walk via `Element.getParentNode()` checking `getAttributeNS()` at each level, following the pattern in `XmlDom.getDefinedNamespaceURI()`. No existing `XmlDom` utility does "find attribute up the hierarchy" for an arbitrary attribute, but the pattern is trivial and can be extracted later.
- 2026-06-06 **Decision**: Unknown `mx:content-mode` values throw `MeshException`, consistent with existing error handling (e.g., `MeshIterator.fromIterationSource()` failures are caught and re-thrown as `MeshException`).
- 2026-06-06 **Finding**: The Markdown-to-DOM pipeline in `MarkdownPageMummifier` preserves `mx:` namespace attributes in embedded HTML blocks: Flexmark passes HTML blocks through as raw text, the XHTML wrapper template is parsed by a namespace-aware `DocumentBuilder`, and `xmlns:mx` declarations on embedded elements are resolved correctly. Adding `xmlns:mx` to the XHTML wrapper template would make `mx:` attributes available document-wide without per-element namespace declarations in Markdown source.
- 2026-06-06 **Open**: Whether to add `xmlns:mx="https://guise.dev/name/mesh/"` to `MarkdownPageMummifier`'s XHTML wrapper template. This is an ergonomic improvement (authors wouldn't need per-element namespace declarations in Markdown-embedded HTML) but leaks the Mesh namespace into the Markdown authoring surface.
- 2026-06-06 **Open**: Future axes not addressed by this ticket: (a) value-converter modes for JS/CSS literal rendering (the Thymeleaf `th:inline="javascript"` equivalent — would be a separate attribute such as `mx:value-format` or similar, not overloaded onto `mx:content-mode`), (b) un-escape / raw-markup insertion (the third orthogonal axis, deliberately kept separate per the Twig `verbatim`/`raw` lesson), (c) a "disable all Mesh processing" mode (Vue `v-pre` equivalent — would be a separate `mx:process="none"` or similar, governing structural directives, not just content).
- 2026-06-06 **Insight**: Guise Mesh processes element content — including `mx:each` clones — while nodes are detached from the live DOM tree. `cloneNode(true)` severs the parent pointer; meshing of the clone runs before reattachment. Any inheritance mechanism that resolves inherited state by walking `getParentNode()` up the live DOM is structurally broken for any cloning directive, because the ancestor holding the inherited state is unreachable during processing.
- 2026-06-06 **Pivot**: The plan initially categorized context-propagation via `MeshContext` as a deferrable performance optimization (O(1) vs O(depth) resolution) and chose an ancestor-walk as simpler. After identifying the detached-clone invariant, the correct characterization is: ancestor-walk-only resolution is broken for `mx:each` clones, not merely slower. Chosen near-term fix: stamp `findEffectiveContentMode(element)` onto each `mx:each` clone before recursing, while the original element is still attached and the mode is resolvable. Context-propagation via `MeshContext` — the correct long-term mechanism, and already the precedent for how `mx:each` iteration variables cross the clone boundary — deferred with a TODO comment.
- 2026-06-06 **Finding**: CommonMark §4.6 type classification for embedded HTML: `<aside>` is a type-6 block (listed by name in the type-6 tag set) and can interrupt a paragraph; a preceding blank line is conventional formatting, not required for block recognition. `<pre>` is a type-1 block (raw-text element) whose end condition is the close tag on a subsequent line, not a blank line. Type-7 blocks (open tags whose names are not in the type-1 or type-6 sets) are the only type that cannot interrupt a paragraph. An earlier plan note that "a blank line is required before `<aside>` for Flexmark to classify it as block-level HTML" was incorrect.
- 2026-06-06 **Milestone**: Confirmed by test that CommonMark type-1 (`<pre>`) and type-6 (`<aside>`) HTML blocks with `xmlns:` declarations on their open tags produce DOM elements with the namespace binding correctly accessible via `hasAttributeNS`/`getAttributeNS` after the Flexmark → XHTML wrapper → namespace-aware `DocumentBuilder` pipeline. The previously untested assumption that namespace propagation survives this pipeline is now covered.
- 2026-06-06 **Insight**: The values `template`/`literal`/`expression` classify *how the content is interpreted* (an ontological category of the text), not *the engine's processing state*. "Mode" belongs to the processing-state end of the semantic spectrum (e.g., strict mode, async mode) and therefore misnames the axis. The discriminating test: the three values share no single action verb — `template` is scanned/interpolated, `expression` is evaluated, `literal` is passed through — so "apply"/"process"/"mode" framings all fail to generalize. The one verb that covers all three is a *reading* verb ("interpret/read as"), which places the axis in the interpretation/classification zone, not the processing zone.
- 2026-06-06 **Decision**: Renamed the attribute from `mx:content-mode` to `mx:content-as`, with "content interpretation" as the design-level axis name. `as` reads as the elision of "interpret/treat/read content as …" and completes naturally with all three values ("content as template/literal/expression"). Rejected alternatives: `content-mode` (misnames the axis as engine state — the original flaw, where the design doc had to strain to explain "mode" meant classification); `content-form` (semantically apt in logic/linguistics but has no foothold as a classifier anywhere in software — "form" appears only frozen in compound terms like normal form / BNF, never standalone, failing the recognizability criterion); `content-kind` (overstates — same characters under different interpretations are not different *kinds* of data); `content-parse`/`content-syntax` (wrong layer — content is already parsed DOM character data, and `^{tree}` has identical syntax whether literal or template); `content-interpretation` (most exact but too long for author-facing markup — kept as the design concept name instead); `content-type` (MIME/media-type baggage).
- 2026-06-06 **Finding**: HTML's own `<link rel="preload" as="script">` attribute is ecosystem precedent for `as` meaning "tell the processor what category to treat this as." Semantic alignment is partial, not exact: HTML `as` names concrete resource/media types (script, font, image), whereas Mesh's values name abstract interpretation strategies. The precedent legitimizes the `as` pattern and the recognizability claim without making the two uses the same concept.
- 2026-06-06 **Decision**: Retained `literal` (not `text`, `raw`, `verbatim`, `plain`, or `static`) for the pass-through value. `literal` is the cross-language industry term for "a value taken at face value" and is the only candidate that is simultaneously semantically correct, grammatically parallel as a noun with `template`/`expression` ("content as a literal"), and free of conflicting axis meaning. Rejected: `text` (all three modes produce text; describes data type, not disposition; collides with existing `mx:text`); `raw` (means un-escape on the serialization axis in Twig, conflating the axes the design separates); `verbatim` (adverb, breaks noun parallelism); `plain` (literal content may still contain elements and directives, so it is not "plain"); `static` (authored template content is equally static; the distinction is interpretation, not static-vs-dynamic).
- 2026-06-06 **Milestone**: `mx:content-as` feature fully implemented in `GuiseMesh`. The `ContentAs` enum (nested in `GuiseMesh`) introduces `TEMPLATE`, `LITERAL`, and reserved `EXPRESSION` values, all round-tripping via `Enums.getSerializationName`/`getSerializedEnum`. `findContentAs(Element)` walks the ancestor chain returning `Optional<ContentAs>`, separating "not present" (inheriting default) from "present but unrecognized" (throws `MeshException` immediately). Unknown values fail at discovery; `EXPRESSION` is recognized but throws "not yet implemented" at use. Clone-stamping preserves the effective interpretation across `mx:each` detachment. 30 passing tests in `GuiseMeshTest` cover all specified cases including two-level nested `mx:each` inheritance.
- 2026-06-06 **Decision**: `ContentAs` values are represented as a nested `enum` in `GuiseMesh` implementing `com.globalmentor.lex.Identifier` rather than as `String` constants. This uses the existing `Enums.getSerializationName`/`getSerializedEnum` facility, which converts `CONSTANT_CASE` ↔ `kebab-case` automatically. The enum switch on `ContentAs` is exhaustive by the compiler, so adding `EXPRESSION` in a future ticket will produce a compile error at every unhandled dispatch site rather than a silent gap. `EXPRESSION` is included in the enum now (reserved) so the compiler enforces completeness from the first implementation.
- 2026-06-06 **Finding**: `XmlDom.setAttribute(Element, NsName, String)` writes a namespace-qualified attribute with no prefix (Javadoc: "Adds a new attribute with no prefix"). In XML, unprefixed attributes are in no namespace — default namespace declarations do not apply to attributes. The clone-stamp therefore produces an attribute node with a non-null namespace URI and a null prefix, a state that cannot arise from parsing well-formed namespaced XML and has no faithful round-trip serialization. This is not an accidental defect: DOM lookup (`getAttributeNS`) is prefix-agnostic and finds the node correctly; and since the stamp attribute is excised before any serialization happens, the anomalous node never reaches the serializer. However, correctness depends on the excision invariant holding. Mitigated by stamping with an explicit prefix via `ATTRIBUTE_CONTENT_AS.withPrefix(NAMESPACE_PREFIX)`, reducing the infoset anomaly; the principled fix remains carrying this state in `MeshContext` rather than the DOM.
- 2026-06-06 **Finding**: The `$` asset prefix is stripped from the output name (`$assets/` → `assets/`) by the capturing group in the default asset name pattern, mirroring the veil `_` prefix behavior. An earlier documentation source claimed the prefix was preserved in output; `DirectoryMummifier.planChildArtifactTargetPath()` is authoritative.
- 2026-06-06 **Finding**: XHTML `<meta property="mummy:...">` CURIE references are normalized from kebab-case to camelCase by `AbstractPageMummifier`'s CURIE resolver before property tag construction. `mummy:alt-location` and `mummy:altLocation` therefore address the same property. This rule applies to all Mummy-namespace `<meta property>` elements, not only `altLocation`.
- 2026-06-06 **Finding**: `architecture.md` omitted `FlangeWebSite` from two exhaustive deploy-target enumerations while the same document cited "Flange environments" as a deploy-target example — an internal contradiction. All four registered targets are now listed.
- 2026-06-06 **Finding**: The blog facility uses two distinct mechanisms: the `@YYYY-MM-DD-slug.md` filename convention (which routes posts into dated URL paths during planning) and the separate `<mummy:directory archetype="blog">` widget (which generates the reverse-chronological listing page). Earlier documentation sources conflated them.
- 2026-06-06 **Lesson**: Framing a documentation review as "update the readmes" and using a practical guide as the gap source anchors the review to already-visible documents. Architecture and design docs end up treated as patch targets rather than corpus peers, causing both factual gaps (FlangeWebSite) and audience-tier inversions (pipeline in readme only) to be missed. The antidote: enumerate the full documentation corpus and map each document's tier and ownership before any generative framing.
