- 2026-03-08 **Decision**: Flange deployment implemented as a `DeployTarget` plugin (configured as `* Flange:` in `deploy.targets`), not as a top-level orchestrator fork in `GuiseMummy.mummify()`. Rejected: top-level `if flangeEnvironment then … else existing targets` — would bypass the plugin architecture and prevent coexistence of legacy and Flange targets.
- 2026-03-08 **Finding**: `globalmentor-application` (which provides `CliStatus`) is not in the `mummy` module's dependency graph — not directly, not transitively. It is only available in the `cli` module. This was incorrectly assumed twice during planning.
- 2026-03-08 **Decision**: Synchronization monitor uses SLF4J logging (parity with existing `S3` deploy target), not `CliStatus`-based progress. Deferred: `CliStatus` progress bar to a future user-feedback/logging/ANSI overhaul. Rejected for now: adding `globalmentor-application` as a `mummy` dependency to avoid coupling the library to CLI infrastructure.
- 2026-03-08 **Milestone**: Initial implementation plan created (`plans/2026-03-08-flange-deploy-target.md`).
- 2026-03-08 **Finding**: `MummyContext` default directory accessors (`getSiteSourceDirectory()` etc.) re-derive paths from configuration on every call via `getProject().getDirectory().resolve(config.getPath(...))`. When config values are absolute (CLI overrides, config-file paths), `Path.resolve(absolutePath)` returns the absolute path unchanged — the project directory's canonicalization is irrelevant. This means CLI and config-file override paths bypass any project directory canonicalization entirely.
- 2026-03-08 **Decision**: `Paths.deriveRealPath(Path, LinkOption...)` added to `globalmentor-core` as the canonicalization primitive. It walks up the ancestor chain until it finds an existing path, calls `toRealPath()` on that, then reattaches non-existent tail segments. This handles the target-directory-doesn't-exist-yet problem uniformly. Rejected: conditional `toRealPath()`-if-exists-else-`.normalize()` fallback — produces paths that are sometimes canonical and sometimes not, so the API contract can't honestly state the result is canonical.
- 2026-03-09 **Insight**: The three site directories (source, target, description target) are essential to the *identity* of the mummy context — the entire artifact tree is rooted in them. They should be constructor parameters, not configuration lookups. The context should validate (via `checkArgumentRealPath()`) but not derive them. This decouples the context from configuration key knowledge and avoids forcing test code to provide fully-populated configurations for paths it never uses.
- 2026-03-09 **Decision**: Introduced `AbstractMummyContext` between `MummyContext` and `BaseMummyContext`. Takes the three directory paths as constructor params (not `Configuration`), validates via `checkArgumentRealPath()`, caches as final fields. Addresses existing TODO on `MummyContext.getRoot()`. Rejected: having `AbstractMummyContext` resolve paths from `Configuration` internally — couples context to config keys and breaks tests that use `Configuration.empty()`.
- 2026-03-09 **Decision**: Named the validation utility `checkArgumentRealPath()` (not `checkArgumentCanonical()`). "Canonical" is already overloaded in Java NIO (`File.getCanonicalPath()` has subtly different semantics from `Path.toRealPath()`). `checkArgumentRealPath` ties directly to the JDK `toRealPath` terminology and avoids the `normalize`/`canonical`/`realPath` confusion.
- 2026-03-09 **Milestone**: Real-path guarantee plan finalized (`plans/2026-03-08-real-path-guarantee.md`). All prerequisites met (`deriveRealPath()`, `checkArgumentRealPath()` in `globalmentor-core`).
- 2026-03-09 **Milestone**: Real-path guarantee implemented across all 8 plan steps. `AbstractMummyContext` created, `MummyContext` default methods removed, `GuiseMummy.createProject()`/`initialize()`/`mummify()` updated, test contexts updated, `StubMummyContext` deleted (dead code). All 87 tests pass.
