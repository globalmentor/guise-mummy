- 2026-03-10 **Finding**: Deploy targets follow a consistent three-phase state model that was previously undocumented: (1) constructor stores immutable configuration as `final` fields and constructs API clients; (2) `prepare()` discovers or provisions infrastructure and caches results as instance fields for later phases (established by `CloudFront.acmCertificateArn`); (3) `deploy()` consumes cached infrastructure state and reads project configuration from `context.getConfiguration()` on demand — it does not use cached configuration. - 2026-03-10 **Finding**: `S3Website.planResource()` generates incorrect redirect targets for collection artifacts. `S3.plan()` walks `comprisedArtifacts()`, which includes the subsumed content artifact (e.g. `foo/index`). Both the `DirectoryArtifact` and its content artifact produce a redirect for the same `altLocation` (via `DirectoryArtifact.getResourceDescription()` delegation). The content artifact's redirect (`foo/bar` → `foo/index`) overwrites the correct directory redirect (`foo/bar` → `foo/`) via map key collision. The surviving redirect targets the non-canonical content path instead of the canonical collection path. Confirmed on a live S3 Website deployment: routing rules redirect to `foo/index` rather than `foo/`. No compensating `**/index` → `**/` canonicalization redirect exists in the deployment code.
- 2026-03-10 **Insight**: The root cause is a conflation of model identity with platform storage. `S3.plan()` walks the content artifact as an independent entity and `S3.planResource()` creates a deploy object attached to it — but the content artifact is an _implementation detail_ for storing collection content, not an independent resource in the site model. Deployers should process the _collection_ artifact as the model entity (using its canonical resource reference for redirect targets and metadata) while deriving the platform storage key and content path from the subsumed content artifact. This is the model-identity vs. platform-storage separation.
- 2026-03-10 **Decision**: Refactor `S3.plan()` and `S3.planResource()` to walk by model identity: skip subsumed artifacts in the tree walk, process each collection artifact once, and pass the content artifact's path separately for storage. `S3ArtifactDeployObject` gains an explicit content `Path` parameter. This fix is scoped to [GUISE-230] because the Flange deployer's `collectRedirects()` must produce correct redirect targets, and the existing S3 code is the reference implementation. Rejected: fixing only in the Flange deployer while leaving the S3 bug in place — would create inconsistent behavior across deploy targets.
- 2026-03-10 **Insight**: The refactoring applies only to `CollectionArtifact`, not all `CompositeArtifact` subtypes. `AspectualArtifact` (e.g. image thumbnails) is also a `CompositeArtifact`, but its comprised artifacts are non-subsumed independent peers — each gets its own deploy object and key. - 2026-03-10 **Finding**: `S3Website` stores S3 object keys in percent-encoded form (e.g. `caf%C3%A9.html` for `café.html`) because the key derivation pipeline flows through `Path.toUri()` → `Artifact.relativizeResourceReference()` → `URIPath.toString()` (which returns `URI.getRawPath()`). - 2026-03-10 **Insight**: Subsumption is a `CompositeArtifact`-level walk invariant, not a `CollectionArtifact`-specific concern. - 2026-03-11 **Finding**: The 2026-03-10 **Finding** about S3 key percent-encoding was wrong. Empirical testing (`S3KeyEncodingInvestigationIT`, 11 tests) proved that `Path.toUri().getRawPath()` on OpenJDK 25 / Windows returns literal non-ASCII characters, not percent-encoded sequences. `URI.create()` silently accepts non-ASCII characters without encoding them, so `getRawPath()` returns whatever was stored — the decoded form. Both Guise Mummy (`Path.toUri()` → `URIPath.relativize()` → `URIPath.toString()`) and Flange (`Path.relativize()` → `objectKeyFromPath()`) produce identical decoded Unicode S3 keys (e.g. `café/index`, not `caf%C3%A9/index`). The claimed encoding divergence does not exist for non-ASCII filenames. A minor divergence exists for spaces (`%20` vs. literal space) because `Path.toUri()` must encode spaces for URI syntax validity. Design document: `designs/s3-key-encoding.md`.
- 2026-03-11 **Lesson**: Reasoning from API documentation (`URI.getRawPath()` "returns the raw, encoded form") without empirical verification led to an incorrect conclusion propagated into the architecture document and minutes. `java.net.URI`'s dual-representation model means "raw" does not guarantee "percent-encoded" — it depends on how the URI was constructed. Any claim about encoding behavior of a multi-step pipeline should be validated with an assertion-based test before being documented as fact.
- 2026-03-11 **Insight**: An S3 object key is an opaque canonical name, not a URI path. The string passed to `PutObject` is stored verbatim — the SDK percent-encodes it for HTTP transport, S3 decodes the HTTP path to recover the original, and the stored key is identical to the caller's input. Therefore the correct value to pass is the canonical resource name (e.g. `café.html`, `my file.html`), never a URI-encoded form. Passing `my%20file.html` would store an object whose key literally contains `%20` — unreachable via normal browser requests because S3 decodes `GET /my%20file.html` to look up key `my file.html`. This makes the space divergence a real bug, not a cosmetic issue.
- 2026-03-11 **Finding**: The central deviation in Guise Mummy's S3 key pipeline is a single call site: `Artifact.relativizeResourceReference(rootTargetPathUri, contentArtifact).toString()`. The `URIPath` carries an inconsistently encoded path — `Path.toUri()` encodes spaces, `#`, `?`, `%` for URI syntax validity, but leaves non-ASCII unencoded (a `java.net.URI` leniency). `.toString()` returns `getRawPath()`, leaking those encoding artifacts into the S3 key. Replacing `.toString()` with `.toDecodedString()` (`URI.getPath()`) recovers the canonical filename. Empirically verified across 20 character classes — spaces, `#`, `%`, non-ASCII, CJK, mixed combinations — `toDecodedString()` always matches the Flange key and the original filesystem name. The round-trip is safe even for filenames containing literal percent sequences (e.g. `caf%C3%A9.html`): `Path.toUri()` encodes `%` as `%25`, then `getPath()` decodes `%25` back to `%`, preserving the original.
- 2026-03-11 **Resolved**: The `toDecodedString()` fix is not merely empirically observed — it is deductively guaranteed by the JDK API contracts. `Path.toUri()` must produce a valid `java.net.URI`; the only character-transformation mechanism available within `URI` is percent-encoding; `URI.getPath()` is contractually defined as decoding all percent-encoded sequences. Since percent-encoding is its own inverse, `getPath()` recovers the original path segment characters regardless of whether the provider encodes minimally or eagerly. `getRawPath()` is encoding-strategy-dependent (diverges across providers); `getPath()` converges. This holds for the relative path segments; the root/drive form is provider-specific but irrelevant to the S3 key derivation.
- 2026-03-11 **Decision**: Chose `toDecodedString()` (one-token fix within existing URI-based architecture) over adopting Flange's `Path.relativize()` + `objectKeyFromPath()` approach (which bypasses URI entirely). Both are contractually correct. The Flange approach avoids URI encoding reasoning altogether, but requires threading a root `Path` through the method signature and introducing a new path-to-key conversion alongside the existing `URIPath`-based resource reference (which is still needed for other purposes at the same call site). Caveat documented: the proof assumes the default filesystem provider; a non-default provider (e.g. in-memory test filesystem or future database-backed filesystem) is not guaranteed by the JDK to fulfill the same round-trip contract.
- 2026-03-11 **Finding**: The S3 key encoding bug affects three call sites, not one: the object key derivation in `S3.plan()`, and both the redirect source key (`altKey`) and redirect target key (`artifactKey`) in `S3Website.planResource()`. All three used `.toString()` (leaking URI-encoding artifacts) and all three were fixed to `.toDecodedString()`.
- 2026-03-11 **Insight**: The bug was invisible for years because it only manifests for characters that `Path.toUri()` percent-encodes for URI syntax validity — space, `#`, `?`, `%`. Non-ASCII characters (which are far more commonly tested) pass through `Path.toUri()` unencoded on OpenJDK/Windows, so `.toString()` and `.toDecodedString()` return identical results for them. The old code was incidentally correct for non-ASCII and would only fail for filenames containing those four URI-significant characters — a class of input unlikely to appear in manual testing.
- 2026-03-11 **Milestone**: S3 key encoding fix applied and verified. Three `.toString()` → `.toDecodedString()` changes across `S3.java` and `S3Website.java`. `S3WebsitePlanIT.testS3KeyIsCanonicalFilesystemName` expanded to 6 character classes (non-ASCII Latin, CJK, space, `#`, `%`, mixed combination) — all green. Design investigation documented in `designs/s3-key-encoding.md`; architecture document updated with dual encoding strategy (`.toString()` for HTML links, `.toDecodedString()` for S3 keys). 87 tests pass, no regressions.
- 2026-03-11 **Insight**: The `S3.plan()` content-finding logic used `CollectionArtifact.getSubsumedArtifacts()` to locate a collection's content artifact, but subsumption does not imply content designation. `CollectionArtifact` represents an abstract collection of child resources — not all collections necessarily provide content via a subsumed artifact. A future `CollectionArtifact` could subsume artifacts for reasons unrelated to content (metadata, indices). `DirectoryArtifact.findContentArtifact()` is the explicit API for "what provides this collection's content," while `getSubsumedArtifacts()` answers "what is not independently deployable" — a different question. Changed `S3.plan()` to use `instanceof DirectoryArtifact` with `findContentArtifact()`, logging a warning for non-`DirectoryArtifact` collections. This ties the deployer to the implementation type; a future `CollectionArtifact.findContentArtifact()` abstraction could generalize it.
