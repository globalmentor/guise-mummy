- 2026-03-11 **Finding**: The 2026-03-10 **Finding** about S3 key percent-encoding was wrong. Empirical testing (`S3KeyEncodingInvestigationIT`, 11 tests) proved that `Path.toUri().getRawPath()` on OpenJDK 25 / Windows returns literal non-ASCII characters, not percent-encoded sequences. `URI.create()` silently accepts non-ASCII characters without encoding them, so `getRawPath()` returns whatever was stored — the decoded form. Both Guise Mummy (`Path.toUri()` → `URIPath.relativize()` → `URIPath.toString()`) and Flange (`Path.relativize()` → `objectKeyFromPath()`) produce identical decoded Unicode S3 keys (e.g. `café/index`, not `caf%C3%A9/index`). The claimed encoding divergence does not exist for non-ASCII filenames. A minor divergence exists for spaces (`%20` vs. literal space) because `Path.toUri()` must encode spaces for URI syntax validity. Design document: `designs/s3-key-encoding.md`.
- 2026-03-11 **Lesson**: Reasoning from API documentation (`URI.getRawPath()` "returns the raw, encoded form") without empirical verification led to an incorrect conclusion propagated into the architecture document and minutes. `java.net.URI`'s dual-representation model means "raw" does not guarantee "percent-encoded" — it depends on how the URI was constructed. Any claim about encoding behavior of a multi-step pipeline should be validated with an assertion-based test before being documented as fact.
- 2026-03-11 **Insight**: An S3 object key is an opaque canonical name, not a URI path. The string passed to `PutObject` is stored verbatim — the SDK percent-encodes it for HTTP transport, S3 decodes the HTTP path to recover the original, and the stored key is identical to the caller's input. Therefore the correct value to pass is the canonical resource name (e.g. `café.html`, `my file.html`), never a URI-encoded form. Passing `my%20file.html` would store an object whose key literally contains `%20` — unreachable via normal browser requests because S3 decodes `GET /my%20file.html` to look up key `my file.html`. This makes the space divergence a real bug, not a cosmetic issue.
- 2026-03-11 **Finding**: The central deviation in Guise Mummy's S3 key pipeline is a single call site: `Artifact.relativizeResourceReference(rootTargetPathUri, contentArtifact).toString()`. The `URIPath` carries an inconsistently encoded path — `Path.toUri()` encodes spaces, `#`, `?`, `%` for URI syntax validity, but leaves non-ASCII unencoded (a `java.net.URI` leniency). `.toString()` returns `getRawPath()`, leaking those encoding artifacts into the S3 key. Replacing `.toString()` with `.toDecodedString()` (`URI.getPath()`) recovers the canonical filename. Empirically verified across 20 character classes — spaces, `#`, `%`, non-ASCII, CJK, mixed combinations — `toDecodedString()` always matches the Flange key and the original filesystem name. The round-trip is safe even for filenames containing literal percent sequences (e.g. `caf%C3%A9.html`): `Path.toUri()` encodes `%` as `%25`, then `getPath()` decodes `%25` back to `%`, preserving the original.
- 2026-03-11 **Resolved**: The `toDecodedString()` fix is not merely empirically observed — it is deductively guaranteed by the JDK API contracts. `Path.toUri()` must produce a valid `java.net.URI`; the only character-transformation mechanism available within `URI` is percent-encoding; `URI.getPath()` is contractually defined as decoding all percent-encoded sequences. Since percent-encoding is its own inverse, `getPath()` recovers the original path segment characters regardless of whether the provider encodes minimally or eagerly. `getRawPath()` is encoding-strategy-dependent (diverges across providers); `getPath()` converges. This holds for the relative path segments; the root/drive form is provider-specific but irrelevant to the S3 key derivation.
- 2026-03-11 **Decision**: Chose `toDecodedString()` (one-token fix within existing URI-based architecture) over adopting Flange's `Path.relativize()` + `objectKeyFromPath()` approach (which bypasses URI entirely). Both are contractually correct. The Flange approach avoids URI encoding reasoning altogether, but requires threading a root `Path` through the method signature and introducing a new path-to-key conversion alongside the existing `URIPath`-based resource reference (which is still needed for other purposes at the same call site). Caveat documented: the proof assumes the default filesystem provider; a non-default provider (e.g. in-memory test filesystem or future database-backed filesystem) is not guaranteed by the JDK to fulfill the same round-trip contract.
- 2026-03-11 **Finding**: The S3 key encoding bug affects three call sites, not one: the object key derivation in `S3.plan()`, and both the redirect source key (`altKey`) and redirect target key (`artifactKey`) in `S3Website.planResource()`. - 2026-03-11 **Insight**: The bug was invisible for years because it only manifests for characters that `Path.toUri()` percent-encodes for URI syntax validity — space, `#`, `?`, `%`. - 2026-03-12 **Decision**: The artifact tree walker is `ArtifactTreeWalker` (a `final` class with static `walk()` methods), not a top-level `ArtifactVisitor` with `Artifact.walk()`. I/O-performing consumers use `throwingConsumer()`/`UncheckedIOException`, with wrapping and unwrapping in the same calling scope. Rejected: `throws IOException` on the visitor — would burden non-I/O consumers for an artifact model that is not inherently I/O-oriented.
