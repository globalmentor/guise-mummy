- 2026-02-13 **Finding**: JEXL 3.3+ introduces a permissions system (JEXL-357) that blocks introspection of application classes by default. Only core JDK packages (`java.lang.*`, `java.util.*`, etc.) are allowed under `RESTRICTED`. Custom `PropertyResolver` implementations bypass the permissions check entirely — they receive objects directly without going through `Introspector` (confirmed by reading JEXL 3.6.2 source code).

- 2026-02-13 **Finding**: Two packages require explicit JEXL introspection permission: `dev.guise.mesh` (for `MeshIterator` accessed via `${iter.current}`) and `dev.guise.mummy` (for `Artifact` and `MummyPlan` injected by `AbstractPageMummifier`). `io.urf.*` does **not** need permission — the custom `URF_PROPERTY_RESOLVER` bypasses standard introspection, so `UrfResourceDescription` access works without permission.

- 2026-02-13 **Decision**: Use `Package...` constructor parameter for `JexlMexlEvaluator` to accept additional permission packages. Rejected alternatives: (1) hardcoded string literals (couples `mesh` module to `mummy` package structure), (2) `String... additionalPackages` (leaks JEXL-specific `.*` wildcard syntax through the API), (3) `JexlPermissions` parameter (breaches abstraction that JEXL is an implementation detail). `Package` is a standard JDK type that conveys semantic intent ("allow this package") without exposing permission expression syntax. Callers use archetypal class references (`Artifact.class.getPackage()`), which are refactoring-safe.

- 2026-02-13 **Insight**: Permission responsibility belongs at the layer that creates/owns the introspected instances. Initial implementation placed `MeshIterator` permission in `JexlMexlEvaluator`, but `GuiseMesh` is what creates `MeshIterator` instances — the evaluator is merely told which classes to permit. Moving base permission to `GuiseMesh` (which prepends `MeshIterator.class` before delegating) clarified that `JexlMexlEvaluator` is a dumb receiver of permissions, not an owner of domain knowledge.

- 2026-02-13 **Decision**: Switch from package-level to class-level permissions using `Set<Class<?>>` + `Set<Package>` constructors on both `GuiseMesh` and `JexlMexlEvaluator`. Supersedes the prior `Package...` decision. Rationale: The set of classes requiring permission is small (5 types: `MeshIterator`, `Artifact`, `MummyPlan`, `CollectionArtifact`, `Mummifier`) and stable — new types only added when new context variables are introduced, which is a significant API change. Class-level permissions via JEXL's `ClassPermissions` provide tighter security (exact canonical name matching) with no maintenance burden. Package-level permissions remain available via `.compose()` for cases where broader access is genuinely needed. `Set` is more idiomatic than varargs for dual-parameter constructors (no array-generics issues).

- 2026-02-13 **Milestone**: JEXL 3.1 → 3.6.2 upgrade complete. Version bumped, permissions configured with `ClassPermissions` for exact class matching, error messages improved (`getMessage()` → `getDetail()`). All 93 tests pass (27 mesh + 66 mummy). Implementation uses modern Java 25 features: pre-`this()` constructor statements (JEP 482) for merging permission sets, and `Stream.concat` + `toUnmodifiableSet()` for concise set composition.
