- 2026-04-01 **Insight**: The aspect fingerprint bug is a structural asymmetry: description saving follows the artifact tree (main + aspects), but description loading follows the source directory walk (main only). Aspect `.description.turf` files are written but never read back — the `DefaultAspectualSourceFileArtifact` constructor copies the main artifact's description as a stopgap, carrying incorrect fingerprints.
- 2026-04-01 **Insight**: In all three responsibilities — description loading, description saving, and aspect discovery — the mummifier is the decision-maker and the artifact is a passive data holder. The gap is that the mummifier loads the main description and saves all descriptions, but never loads aspect descriptions. The fix is to close the asymmetry by having the mummifier load aspect descriptions too.
- 2026-04-01 **Decision**: The mummifier (`DefaultImageMummifier.createArtifact()`) will build aspect artifacts fully — with individually loaded descriptions — before creating the main artifact. This follows the existing pattern in `DirectoryMummifier`, where children are created before the parent. Rejected: passing aspect specs (paths + descriptions) through the builder; providing a callback interface for description loading; having the artifact constructor load descriptions itself.
- 2026-04-01 **Decision**: Reuse `AbstractFileMummifier.loadArtifactDescription(context, sourceFile, targetFile)` for aspects, passing the main image as `sourceFile` and the aspect's target path as `targetFile`. The main image is semantically the true source of the aspect, and the method's timestamp and metadata loading logic produces correct results in all scenarios (cached/uncached, clean/dirty).
- 2026-04-01 **Finding**: The `removeProperty(Content.MODIFIED_AT_PROPERTY_TAG)` workaround in `DefaultImageMummifier.mummifyFile()` serves two purposes: compensating for wrong descriptions (fixed by loading correct descriptions) and forcing regeneration when the parent changes (still needed because aspects don't go through `plan()` and have no source-change detection). Even with correct descriptions, a separate mechanism is needed to force aspect mummification when the parent is dirty.
- 2026-04-02 **Decision**: The `invariably` parameter goes on the `Mummifier` interface, not just `AbstractFileMummifier`. Incremental mummification is a framework-level concept with established industry semantics (Make, Gradle, Maven), not an implementation detail of file-based mummifiers. The existing `mummify(context, artifact)` becomes a `default` method delegating with `invariably = false`. Rejected: `protected` overload on `AbstractFileMummifier` only (prevents other mummifier types from participating); `isDirty` naming (specific to timestamp-checking model, not mechanism-neutral).
- 2026-04-02 **Insight**: `Mummifier.mummify()` conflates two operations: "ensure this artifact is up to date" (orchestration/gating) and "generate this artifact's output" (the actual transformation). The `invariably` parameter addresses this by letting the caller say "the generation step must happen" while still going through all orchestration (directory creation, description saving, fingerprint computation). This mirrors the internal `mummifyFile()` separation, which already documents itself as "invariably mummifies."
- 2026-04-02 **Decision**: `invariably` propagates to comprised artifacts: a mummifier may escalate to `true` for a child but must not downgrade `true` to `false`. This makes `invariably` equivalent to scoped `--full` for a subtree. Rejected: treating `invariably` as applying only to the gating decision of the receiving mummifier, with no transitive semantics.
- 2026-04-02 **Milestone**: Core fix implemented. Aspect descriptions are now individually loaded via `loadArtifactDescription()` in `DefaultImageMummifier.createArtifact()`. The `removeProperty(MODIFIED_AT)` workaround is replaced by `mummify(context, aspectArtifact, true)`. The `DefaultAspectualSourceFileArtifact` constructor is reduced to a single-arg builder pull.
- 2026-04-02 **Open**: The existing `TODO set aspect ID` (originally on the builder `.build()` call) likely intended a first-class aspect-ID field on the artifact or builder, rather than relying solely on a description property. Preserved verbatim for future consideration.
- 2026-04-02 **Milestone**: Fix manually verified against the production project that originally surfaced the bug.
